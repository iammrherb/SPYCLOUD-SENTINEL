Data Flow Details
How data moves from SpyCloud APIs through to Sentinel incidents and automated response
  1. SpyCloud API Polling — Function App polls 7 SpyCloud API products on configurable schedule (default: every 6 hours)
  2. Data Ingestion — Raw breach/exposure data sent to Data Collection Endpoint via DCR
  3. Table Storage — Data lands in 11 custom Log Analytics tables (SpyCloudBreachWatchlist_CL, etc.)
  4. Detection — 38 analytics rules run scheduled KQL queries against the tables
  5. Incident Creation — Matching rules create Sentinel incidents with entity mappings
  6. Automation — Automation rules trigger playbooks based on incident properties
  7. Remediation — Playbooks execute actions: password reset, device isolation, MFA enforcement, etc.
  8. Enrichment — AI Engine adds threat context, executive summaries, and remediation plans

Connector Configuration
Required settings for the data connector

Required Parameters

  • SpyCloud API Key — Enterprise API key from SpyCloud portal
  • Monitored Domains — Comma-separated list of domains to watch
  • Workspace ID — Log Analytics workspace ID
  • Workspace Key — Log Analytics shared key

Optional Settings

  • Polling Interval — Default: 6 hours (configurable)
  • Severity Threshold — Minimum severity for alerts (default: 2)
  • API Products — Select which products to enable
  • Batch Size — Records per API call (default: 1000)

CloudShell Deployment
# 1. Open Azure CloudShell (bash) # 2. Clone the repository git clone https://github.com/iammrherb/SPYCLOUD-SENTINEL.git cd SPYCLOUD-SENTINEL # 3. Run the deployment script chmod +x scripts/deploy-cloudshell.sh ./scripts/deploy-cloudshell.sh \ --resource-group "your-rg" \ --workspace "your-sentinel-workspace" \ --spycloud-api-key "your-api-key" \ --domains "yourdomain.com,other.com" # 4. Run post-deployment configuration ./scripts/post-deploy.sh \ --resource-group "your-rg" \ --workspace "your-sentinel-workspace"

SpyCloud Requirements

  • SpyCloud Enterprise API key
